《創世者：白紙宇宙》開發計劃書 (6-Month MVP Roadmap)
📌 技術棧確認 (Technology Stack)
前端/客戶端： Unity 3D (C#) - 負責渲染、UI、跨平台打包 (iOS/Android)。
後端/大腦： Python (FastAPI) - 處理業務邏輯、計費、對接 AI API。
多人連線服務器： Photon Fusion 或 Nakama - 處理玩家位置與物品狀態同步。
AI 服務矩陣：
文本轉 3D：Meshy API / CSM.ai API
安全審核與文本優化：OpenAI API (GPT-4o-mini + Moderation)
資料庫與儲存：
關聯/向量庫：PostgreSQL (pgvector) (用於查找相似物品，實現緩存)。
雲端儲存：Cloudflare R2 (存儲 3D 模型，免流出流量費)。
🗓️ 階段開發時間表 (Phased Timeline)
第一階段：核心造物循環 (Month 1 - 單機原型)
目標： 跑通「輸入文字 -> 扣除魔法值 -> 生成 3D 模型 -> 渲染到遊戲中」的閉環。
W1-W2 (前端+後端)：
搭建 Unity 基礎純白場景、玩家虛擬搖桿移動。
開發「配方式」創造 UI（材質、風格、物品 3 個下拉框/短文本框）。
架設 FastAPI 後端基礎框架，打通前端與後端的 API 通訊。
W3 (AI 接入)：
後端接入 OpenAI Moderation 進行違禁詞攔截（測試拒絕生成邏輯）。
後端接入 Text-to-3D API，獲取生成的 .glb 文件並上傳至雲端儲存。
W4 (渲染與優化)：
Unity 客戶端實現動態下載下載 .glb 文件並在玩家面前加載（使用 glTFast 插件）。
添加「魔法陣生成中」的 15 秒等待特效，掩蓋 AI 生成的延遲。
第二階段：降本增效與資料庫 (Month 2 - 後端強化)
目標： 建立緩存系統（省錢大招）與基礎帳號系統。
W1-W2 (緩存與向量匹配)：
後端配置 PostgreSQL 向量資料庫。
當玩家提交 Prompt 時，先轉換為向量比對歷史記錄。如果相似度 > 90%，直接返回已存在的雲端模型 URL（不呼叫生成 API）。
W3 (帳號與數值)：
實現玩家註冊/登入（建議接入 Google/Apple 授權登入）。
後端建立【創世塵埃 (Mana)】的扣除邏輯與每日免費發放邏輯。
W4 (地塊系統)：
Unity 內實現網格系統 (Grid System)。
玩家可以將生成的物品進行旋轉、擺放，並將坐標數據保存到伺服器。
第三階段：多人同屏與世界同步 (Month 3 - 網遊化)
目標： 把單機建造變成大型多人在線遊戲 (MMO)。
W1-W2 (連線接入)：
接入 Photon Fusion 或 Nakama。
實現多個玩家在同一個純白場景中看到彼此的 Avatar（虛擬形象）和移動。
W3-W4 (狀態同步)：
實現「當玩家 A 放下一張桌子，玩家 B 手機上也能實時加載並看到這張桌子」。
技術難點攻堅： 動態加載 LOD（細節層次）優化。確保同畫面出現 50 個 AI 生成物品時，手機不會過熱閃退。
第四階段：經濟系統與商業化 (Month 4 - 賺錢模塊)
目標： 實現雙代幣經濟與應用商店內購，為上架做準備。
W1-W2 (應用內購 IAP)：
註冊 Apple Developer 和 Google Play Console 帳號。
在 Unity 中接入 IAP SDK，實現真實信用卡充值購買【創世塵埃 (Mana)】，並與後端數據同步。
W3 (星幣與交易市場)：
開發遊戲內二手市場 (Bazaar) UI。
實現玩家上架物品、設定價格（星幣）。
實現玩家之間的星幣交易，及系統自動抽成 10% 稅率的後端邏輯。
W4 (防作弊與日誌)：
增加交易日誌記錄。防範前端修改內存數據偽造魔法值的作弊行為（所有數值計算必須在後端進行）。
第五階段：合規審查與打磨 (Month 5 - 上架前置)
目標： 滿足 Apple 和 Google 的嚴苛審核要求，優化體驗。
W1 (合規功能必做)：
加入「舉報按鈕」：玩家可以舉報場景中的不雅模型。
加入「拉黑名單」：玩家可以屏蔽某些人生成的所有內容。
完善隱私政策 (Privacy Policy) 和用戶協議 (TOS)。
W2-W3 (新手引導與留存)：
製作開場的 NPC 新手教學（引導玩家生成第一個物品）。
開發「排行榜系統」和「萬物圖鑑系統」（點亮收集物品獲得獎勵）。
W4 (QA 內部測試)：
TestFlight (iOS) / 內部測試軌道 (Android) 分發給 20-50 名親友玩家進行壓力測試。
監測 API 實際消耗成本與玩家充值意願的比率。
第六階段：審核與發布 (Month 6 - 上線)
目標： 順利上架，開啟運營。
W1-W2 (提審攻堅)：
準備應用商店的截圖、宣傳影片（展示玩家輸入文字變出 3D 物品的酷炫過程）。
提交 App Store 和 Google Play 審核。(註：涉及 AI 和 UGC 的 App 通常會被拒絕 1-2 次，需根據審核員反饋解釋 Moderation API 和舉報機制)。
W3 (軟啟動 Soft Launch)：
在單一地區（如加拿大或東南亞）先行上架測試數據，觀察留存率（D1, D7）和付費率（LTV）。
W4 (全球發布與營銷)：
正式全球上架。
在 TikTok/YouTube Shorts 發佈短影音行銷（主打「用嘴做遊戲」、「AI 建造世界」的標籤）。
💰 預算與成本控制建議 (Cost Management)
前期開發成本（大頭是人力）：
盡量使用 Unity Asset Store 的現成 UI 插件和基礎代碼庫，減少重複造輪子。
開發期設定 API 消耗上限（Hard Limit），防止代碼死循環導致 OpenAI / Meshy 帳單爆炸。
營運期成本策略：
定價策略： 如果生成一個 3D 模型 API 成本是 $0.05，那麼你在遊戲內消耗的 Mana 價值必須標價至少 $0.15 ~ $0.25。
緩存率目標： 爭取上線一個月後，玩家 40% 的創造指令都能從資料庫直接拉取緩存（0 成本），這將是你的暴利來源。
🛡️ 風險預警與對策
3D 模型質量不可控： AI 生成的 3D 模型可能有時會破圖或材質奇怪。
對策： 在遊戲美術設定上，將這些「瑕疵」包裝成「宇宙初創時的不穩定量子態」，用世界觀去合理化技術缺陷。
手機性能崩潰： 玩家在一個區域堆了幾百個高精度模型。
對策： 設置領地「承載力」上限。每個模型根據多邊形數量佔用不同的承載力，滿了就不允許放置。
