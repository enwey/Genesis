# 遊戲開發專案計畫書：Genesis - The Blank Canvas (創世者：白紙宇宙)

**文件版本:** v1.0
**專案類型:** AIGC 沙盒建造 / MMO 虛擬經濟
**目標平台:** iOS / Android (MVP階段), 後續拓展 PC
**預估 MVP 開發週期:** 6 個月

---

## 1. 專案概述 (Project Overview)
一款以「文字生成 3D 萬物」為核心驅動的多人線上沙盒遊戲。玩家降生於無限大的純白網格世界，透過消耗付費代幣輸入提示詞（Prompt），由 AI 即時生成 3D 資產進行建造，並在玩家間形成自由交易的虛擬經濟體。

---

## 2. 技術架構選型 (Technology Stack)

| 模塊 | 技術/工具 | 說明 |
| :--- | :--- | :--- |
| **前端引擎** | Unity 3D (C#) | 支援跨平台打包，具備強大的動態模型加載外掛 (`glTFast`) |
| **後端框架** | Python (FastAPI) | 處理核心業務邏輯、計費，最適合與 AI 大模型 API 對接 |
| **多人連線** | Photon Fusion / Nakama | 處理玩家實時位置與場景物品狀態同步 |
| **AI 文本審核**| OpenAI Moderation API | **必備**：攔截違禁詞彙，確保 App Store / Google Play 過審 |
| **AI 3D 生成**| Meshy API / CSM.ai | 將玩家文本轉換為 `.glb` 低多邊形 3D 模型 |
| **向量資料庫** | PostgreSQL (pgvector) | 語義搜索歷史 Prompt，實現模型緩存，大幅降低 API 成本 |
| **雲端儲存** | Cloudflare R2 / AWS S3 | 儲存 AI 生成的 3D 模型文件 (推薦 R2，免流出流量費) |

## 3. 核心經濟與合規設計 (Business & Compliance)

- **雙代幣經濟 (規避法律風險)：**
  - **【創世塵埃 (Mana)】：** 法幣充值貨幣 (IAP)，唯一用途是驅動 AI 造物。**開發者核心利潤來源。**
  - **【星幣 (StarCoin)】：** 遊戲內產出/交易貨幣，不可提現。用於玩家間買賣物品。**開發者抽取 10% 交易稅。**
- **防成本倒掛機制：** 採用「配方填空式」UI 限制 Prompt 長度，並根據生成複雜度動態扣除 Mana。
- **內容合規機制：** 強制 AI 文本審核 + 玩家實體物品舉報/拉黑系統。

## 4. 六個月 MVP 開發里程碑 (6-Month Roadmap)

### Phase 1: 核心造物閉環 (Month 1)
**目標：** 跑通從輸入文字到生成 3D 模型的單機流程。
- [ ] 建立 Unity 基礎專案，實現純白網格場景與虛擬搖桿移動。
- [ ] 開發「配方式」創造 UI (包含：材質、風格、物品類型的下拉選單/文本框)。
- [ ] 架設 FastAPI 後端基礎框架。
- [ ] 後端串接 OpenAI Moderation API 進行違規詞攔截測試。
- [ ] 後端串接 3D 生成 API (如 Meshy)，並將生成的 `.glb` 上傳至雲端儲存。
- [ ] 前端實作動態下載 `.glb` 檔案並在運行時渲染進場景中。
- [ ] 新增「魔法陣編織中」的 15 秒等待特效，掩蓋 AI 生成延遲。

### Phase 2: 降本增效與系統完善 (Month 2)
**目標：** 建立向量緩存系統以節省 API 成本，並實作帳號系統。
- [ ] 後端部署 PostgreSQL 配合 `pgvector` 外掛。
- [ ] 實作緩存邏輯：提交 Prompt 前進行向量比對，相似度 > 90% 則直接回傳舊有雲端模型 URL (免扣 API 費)。
- [ ] 實作玩家帳號註冊/登入 (支援 Apple/Google 授權登入)。
- [ ] 後端實作【創世塵埃 (Mana)】的扣除邏輯與每日發放邏輯。
- [ ] Unity 端實作建造網格系統 (Grid System)，允許玩家旋轉、擺放模型並儲存座標。

### Phase 3: 多人連線與網遊化 (Month 3)
**目標：** 實現玩家同屏互動與世界狀態同步。
- [ ] 整合 Photon Fusion 或 Nakama 網路 SDK。
- [ ] 實作玩家 Avatar 跨客戶端的位置與動作同步。
- [ ] 實作場景物品同步 (當玩家 A 放置物品，玩家 B 畫面即時加載並顯示)。
- [ ] 技術攻堅：實作客戶端 LOD (細節層次) 與遮擋剔除，防止模型過多導致手機閃退。

### Phase 4: 商業化與經濟體系 (Month 4)
**目標：** 建立應用內購與玩家間交易市場。
- [ ] 註冊 Apple Developer 與 Google Play Console 帳號。
- [ ] Unity 整合 IAP (應用內購) SDK，實作真實法幣購買 Mana 流程。
- [ ] 開發「星幣 (StarCoin)」系統與伺服器數值驗證。
- [ ] 開發拍賣行 / 二手市場 UI (The Bazaar)。
- [ ] 實作玩家上架物品、設定價格、完成交易及系統抽成 10% 的完整後端邏輯。

### Phase 5: 合規、優化與測試 (Month 5)
**目標：** 滿足雙端應用商店審核標準，準備上架。
- [ ] **強制合規：** 實作遊戲內「舉報物品/玩家」與「拉黑名單」功能。
- [ ] 撰寫並實裝隱私權政策 (Privacy Policy) 與服務條款 (TOS)。
- [ ] 製作 NPC 新手教學流程 (Onboarding)。
- [ ] 開發「萬物圖鑑系統」與「世界建築點讚排行榜」。
- [ ] 進行 TestFlight / Google 內部測試軌道封測 (20-50人)，監測效能與真實 API 成本。

### Phase 6: 審核與發行 (Month 6)
**目標：** 解決審核問題，正式營運。
- [ ] 準備商店宣傳素材 (截圖、展示 AI 變物過程的行銷影片)。
- [ ] 提交 App Store 與 Google Play 審核 (預留被拒絕及修改說明的時間)。
- [ ] 於單一測試地區 (Soft Launch) 上架，觀測 D1/D7 留存率與付費轉換率。
- [ ] 根據數據調整 Mana 售價與 API 緩存閾值。
- [ ] 全球正式發行，啟動社群行銷 (TikTok/YouTube Shorts)。

## 5. 成本與風險管理 (Risk Management)

### 成本控制
1. **API 防刷機制：** 嚴格在伺服器端驗證 Mana 餘額，防範封包竄改無限生成。開發期間對 AI API 設定硬性配額 (Hard Limit)。
2. **利潤定價法：** 遊戲內 Mana 售價需定於實際 API 平均成本的 3~5 倍以上。
3. **緩存指標：** 上線後首要營運目標是將「歷史模型緩存命中率」提升至 40% 以上，此為純利潤來源。

### 潛在風險與對策
1. **AI 模型破圖/品質不穩：** 
   * **對策：** 利用世界觀包裝 (如「量子不穩定現象」)；提供低成本的「重鑄」功能。
2. **審核因 UGC 內容被拒：** 
   * **對策：** 提審時向審核員提供詳細的 OpenAI Moderation 攔截日誌範例，強調嚴格的舉報機制。
3. **客戶端效能崩潰：** 
   * **對策：** 設定個人領地「承載力」上限 (Capacity Limits)，高多邊形模型佔用更多承載力，滿額禁止放置。
